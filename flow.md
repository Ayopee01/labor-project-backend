# เอกสาร Flow การทำงานของระบบจัดการแรงงานลงสินค้า

เอกสารนี้เป็นเอกสารอ้างอิงการทำงานของระบบในเชิงธุรกิจและเชิงระบบเท่านั้น ไม่ใช่เอกสารสถานะการพัฒนา ไม่อธิบายว่า Backend, Frontend หรือ Mobile ทำไปถึงจุดใดแล้ว และไม่ใช้เป็น checklist implementation

วัตถุประสงค์ของเอกสารนี้คือทำให้ทุกฝ่ายเข้าใจ flow เดียวกันตั้งแต่รถเข้าประตูตลาด การสร้างงาน การเรียกคนงาน การสแกน QR การลงสินค้า การส่งยอด การยืนยันจากร้านค้า การยกเลิก การหมดเวลา และการปิดงาน

---

## 1. หลักการของระบบ

### 1.1 หน่วยงานหลักของระบบ

ระบบจัดการงานลงสินค้าโดยยึดรถเป็นศูนย์กลาง

1. `VehicleJob` คืองานหลักระดับรถ 1 คัน
2. `MarketJob` คืองานย่อยระดับตลาดหรือเลขที่งานตลาด 14 หลัก
3. `GateTicket` คือตั๋วลงสินค้า 1 ใบ ซึ่งแทน 1 แผงค้า
4. `TicketProduct` คือสินค้าในตั๋วหรือแผงนั้น
5. `VehicleJobAssignment` คือการมอบหมายคนงานให้รถ 1 คัน
6. คนงาน 1 ชุดถูกเรียกสำหรับรถ 1 คัน และรับผิดชอบทุกตลาด ทุกแผง และทุกตั๋วในรถคันนั้นจนจบงาน

### 1.2 กฎสำคัญของระบบ

1. รถ 1 คันมีงานหลักระดับรถ 1 งาน
2. รถ 1 คันมีได้หลายตลาด
3. ตลาด 1 ตลาดมีได้หลายแผง
4. ตั๋ว Gate 1 ใบมีได้เพียง 1 แผง
5. ถ้ารถคันเดียวกันมีหลายแผงในตลาดเดียวกัน ตั๋วหลายใบสามารถใช้เลขที่งานตลาด 14 หลักเดียวกันได้
6. ถ้ารถคันเดียวกันไปหลายตลาด จะมี `MarketJob` หลายรายการอยู่ใต้งานรถเดียวกัน
7. คนขับใช้ QR ระดับรถ
8. คนงานใช้ QR ชุดกลางของรถ โดยสแกนจากตั๋วใบใดก็ได้ในชุดนั้น
9. จำนวนคนงานที่ต้องใช้ (`workersRequired`) อยู่ระดับรถ
10. ระบบคำนวณ `workersRequired` จากแผงที่ต้องใช้คนงานมากที่สุด ไม่ใช่นับรวมสินค้าทุกแผงทั้งรถ
11. เมื่องานเริ่มลงสินค้าแล้ว ทุกตลาดและทุกตั๋วที่ยังไม่จบในรถคันเดียวกันเข้าสู่สถานะ `IN_PROGRESS` พร้อมกัน
12. ร้านค้า/แม่ค้ายืนยันยอดแยกระดับตั๋วหรือแผง
13. ตลาดปิดเมื่อทุกตั๋วในตลาดนั้นจบ
14. รถปิดเมื่อทุกตลาดและทุกตั๋วในรถนั้นจบ
15. คนงานคืนคิวเมื่อรถทั้งคันจบ หรือเมื่อถูกยกเลิกตามกฎของระบบ

## 2. บทบาทผู้ใช้งานและระบบที่เกี่ยวข้อง

### 2.1 Gate

Gate เป็นจุดเริ่มต้นของงาน มีหน้าที่ส่งข้อมูลรถและตั๋วเข้าสู่ระบบ

Gate ส่งข้อมูลหลัก:

- ทะเบียนรถ
- รหัสอ้างอิงธุรกรรม
- ตั๋วลงสินค้าทั้งหมดของรถ
- เลขที่งานตลาด 14 หลัก
- ตลาดปลายทาง
- แผงปลายทาง
- รายการสินค้าและจำนวนที่คาดไว้
- ข้อมูลร้านค้า/แม่ค้า เช่น LINE ID ถ้ามี
- สถานะเริ่มต้นของรถ ถ้ามี

Gate รับข้อมูลกลับเพื่อใช้พิมพ์ตั๋ว:

- รหัสงานรถ
- จำนวนคนงานที่ต้องใช้
- ข้อมูลประกอบการคำนวณจำนวนคนงาน
- QR คนขับ
- QR คนงานหรือ QR ชุดกลาง
- mapping ระหว่างตั๋ว Gate กับรหัสภายในระบบ
- payload สำหรับพิมพ์ตั๋ว

### 2.2 คนขับ

คนขับมีหน้าที่ดูข้อมูลงานรถจาก QR และกดพร้อมลงเมื่อรถและปลายทางพร้อม

หน้าที่หลัก:

- สแกน QR ระดับรถ
- ตรวจรายละเอียดตลาดและแผงปลายทาง
- รอความพร้อมของแผงหรือร้านค้า
- กด `พร้อมลง` หรือ `ลงเลย` เมื่อพร้อม
- ติดตามสถานะคิวหรือสถานะเรียกคนงาน

### 2.3 คนงาน

คนงานมีหน้าที่รับงานรถ ไปยังพื้นที่ลงสินค้า สแกน QR ชุดกลาง ลงสินค้า ตรวจนับ และส่งยอด

หน้าที่หลัก:

- Login เข้าแอป
- กด Online เพื่อเข้าคิว
- รับหรือปฏิเสธงานภายในเวลาที่กำหนด
- สแกน QR คนงานภายในเวลาที่กำหนดหลังรับงาน
- ลงสินค้าให้ทุกตลาดและทุกแผงของรถที่ได้รับมอบหมาย
- ส่งจำนวนสินค้าจริงแยกตามตั๋ว/แผง
- แก้ไขยอดหากร้านค้าปฏิเสธ

### 2.4 ร้านค้า/แม่ค้า/Vendor

ร้านค้าเป็นผู้ยืนยันยอดสินค้าที่คนงานส่งมาในระดับแผง

หน้าที่หลัก:

- รับแจ้งว่ารถมาถึงหรือมีงานรอ
- รับยอดสินค้าที่คนงานส่ง
- ยืนยันยอดแผงของตน
- ปฏิเสธยอดพร้อมเหตุผลเมื่อต้องแก้ไข
- รับแจ้งค่าบริการหรือข้อมูลหลังปิดงานตาม flow ที่กำหนด

### 2.5 Admin

Admin ดูแลภาพรวม คิวรถ คิวคนงาน และจัดการเหตุการณ์ผิดปกติ

หน้าที่หลัก:

- ดูคิวรถและสถานะงาน
- ดูคิวคนงานและสถานะคนงาน
- เปลี่ยนรถกลับไปรอ
- ยกเลิกงานทั้งรถ
- ยกเลิกเฉพาะตลาด
- ยกเลิกเฉพาะตั๋ว/แผง
- ยกเลิกคนงานรายคน
- ต่อเวลา scan QR
- assign คนงานแบบ manual
- บันทึกเหตุผลเมื่อทำ action สำคัญ

## 3. สถานะหลักของงาน

### 3.1 สถานะงานรถและงานตลาด

| สถานะ | ความหมาย |
|---|---|
| `WAIT` | รอความพร้อม รถหรือปลายทางยังไม่พร้อมลง |
| `DISPATCH_NOW` | พร้อมเรียกคนงานหรือกำลังรอคนงานว่าง |
| `IN_PROGRESS` | คนงานสแกนครบและเริ่มลงสินค้าแล้ว |
| `COMPLETED` | งานจบครบตามเงื่อนไข |
| `CANCELLED` | งานถูกยกเลิก |
| `CALLED` | สถานะเดิมสำหรับบาง flow เก่า ไม่ใช่สถานะหลักของ flow ปัจจุบัน |

### 3.2 สถานะตั๋ว/แผง

| สถานะ | ความหมาย |
|---|---|
| `WAIT` | ตั๋ว/แผงยังรอความพร้อม |
| `READY` | ตั๋ว/แผงพร้อมลงสินค้า |
| `IN_PROGRESS` | กำลังลงสินค้าหรือยังอยู่ในงานที่เริ่มแล้ว |
| `SUBMITTED` | คนงานส่งยอดแล้ว รอร้านค้ายืนยัน |
| `NEEDS_REVISION` | ร้านค้าปฏิเสธยอด ต้องแก้ไข |
| `COMPLETED` | ร้านค้ายืนยันแล้ว ตั๋ว/แผงจบ |
| `CANCELLED` | ตั๋ว/แผงถูกยกเลิก |

### 3.3 สถานะการยืนยันของร้านค้า

| สถานะ | ความหมาย |
|---|---|
| `NOT_SUBMITTED` | ยังไม่มีการส่งยอด |
| `PENDING` | ส่งยอดแล้ว รอร้านค้ายืนยัน |
| `CONFIRMED` | ร้านค้ายืนยันยอดแล้ว |
| `REJECTED` | ร้านค้าปฏิเสธยอด |
| `CANCELLED` | ตั๋ว/แผงถูกยกเลิก |

### 3.4 สถานะ Assignment คนงาน

| สถานะ | ความหมาย |
|---|---|
| `PENDING` | งานถูกส่งให้คนงานแล้ว รอรับงาน |
| `ACCEPTED` | คนงานรับงานแล้ว รอสแกน QR |
| `REJECTED` | คนงานปฏิเสธงาน |
| `TIMEOUT` | คนงานไม่ตอบรับหรือไม่สแกนภายในเวลา |
| `SCANNED` | คนงานสแกน QR สำเร็จแล้ว |
| `COUNTING` | คนงานส่งยอดแล้วหรือกำลังอยู่ในรอบแก้ไขยอด |
| `COMPLETED` | Assignment เสร็จแล้ว |
| `CANCELLED` | Assignment ถูกยกเลิก |

### 3.5 สถานะคนงานในคิว

| สถานะ | ความหมาย |
|---|---|
| `offline` | ไม่อยู่ในคิว ไม่รับงาน |
| `ready` | พร้อมรับงาน อยู่ใน FIFO queue |
| `busy` | มีงานหรืออยู่ในกระบวนการ assignment |
| `break` | พักงานชั่วคราว |

## 4. Flow หลักตั้งแต่รถเข้าถึงปิดงาน

### 4.1 รถเข้าประตูตลาด

1. รถบรรทุกมาถึง Gate
2. Gate ตรวจข้อมูลรถและตั๋ว
3. Gate ส่งข้อมูลรถและตั๋วทั้งหมดเข้าสู่ระบบ
4. ระบบตรวจว่า request นี้เป็นธุรกรรมใหม่หรือเป็น retry ของธุรกรรมเดิม
5. ถ้าเป็นธุรกรรมซ้ำ ระบบต้องตอบกลับข้อมูลเดิม ไม่สร้างงานซ้ำ
6. ถ้าเป็นธุรกรรมใหม่ ระบบสร้างงานรถและงานย่อยทั้งหมด

### 4.2 การสร้างงานรถ

ระบบสร้างโครงสร้างงานดังนี้:

```text
VehicleJob
  ├─ MarketJob A
  │   ├─ GateTicket A1
  │   │   ├─ TicketProduct 1
  │   │   └─ TicketProduct 2
  │   └─ GateTicket A2
  │       └─ TicketProduct 3
  └─ MarketJob B
      └─ GateTicket B1
          └─ TicketProduct 4
```

กฎการสร้าง:

1. รถ 1 คันสร้าง `VehicleJob` เดียว
2. ตั๋วที่มีเลขที่งานตลาดเดียวกันอยู่ใต้ `MarketJob` เดียวกัน
3. ตั๋วคนละเลขที่งานตลาดหรือคนละตลาดแยกเป็น `MarketJob` คนละรายการ
4. ตั๋ว 1 ใบผูกกับ 1 แผงเท่านั้น
5. สินค้าในตั๋วถูกเก็บเป็นรายการแยก

### 4.3 การคำนวณจำนวนคนงาน

ระบบคำนวณจำนวนคนงานจากยอดสินค้ารวมภายในแต่ละแผง

ตัวอย่าง rule:

| จำนวนสินค้ารวมในแผงเดียว | จำนวนคนงานของแผง |
|---|---:|
| 1-100 | 1 |
| 101-150 | 2 |
| มากกว่า 150 | 3 |

จากนั้นระบบเลือกค่าที่มากที่สุดของทุกแผงเป็น `workersRequired` ระดับรถ

ตัวอย่าง:

| แผง | จำนวนรวม | คนงานที่แผงต้องใช้ |
|---|---:|---:|
| A-01 | 80 | 1 |
| A-02 | 130 | 2 |
| B-01 | 40 | 1 |

ผลลัพธ์ระดับรถคือ `workersRequired = 2`

ระบบไม่รวมยอดสินค้า A-01 + A-02 + B-01 เพื่อคำนวณคนงาน เพราะคนงานชุดเดียวทำงานเป็นชุดต่อรถ และจำนวนคนงานใช้ตามแผงที่หนักที่สุด

### 4.4 การกำหนดสถานะเริ่มต้น

ถ้า Gate ส่งสถานะเริ่มต้นมา ระบบใช้สถานะนั้นตามกฎที่อนุญาต

ถ้า Gate ไม่ส่งสถานะเริ่มต้น ระบบประเมินความพร้อม:

1. ถ้ารถและแผงพร้อมลง ให้รถและตลาดเข้าสู่ `DISPATCH_NOW`
2. ถ้ายังไม่พร้อม ให้รถและตลาดเข้าสู่ `WAIT`

สถานะก่อนเริ่มลงสินค้าของทุกตลาดในรถเดียวกันต้องแสดงสอดคล้องกัน เช่น:

- ทุกตลาดเป็น `รอลง`
- ทุกตลาดเป็น `ลงเลย`
- ทุกตลาดเป็น `รอคิวลง`

### 4.5 การตอบกลับ Gate

ระบบส่งข้อมูลกลับให้ Gate เพื่อพิมพ์ตั๋ว:

1. รหัสงานรถ
2. `workersRequired`
3. ข้อมูลการคำนวณคนงาน
4. QR คนขับระดับรถ
5. QR คนงานชุดกลาง
6. รายการตลาด ตั๋ว แผง และสินค้า
7. payload สำหรับพิมพ์ตั๋ว

## 5. Flow คนขับ

### 5.1 คนขับเปิดงานจาก QR

1. คนขับสแกน QR ระดับรถ
2. ระบบตรวจ token ของ QR
3. ระบบโหลดข้อมูลรถ ตลาด ตั๋ว แผง และสินค้า
4. หน้า Driver Web แสดงข้อมูลงาน
5. คนขับไม่ต้องกรอก `jobId`
6. ระบบ derive งานจาก QR token เท่านั้น

### 5.2 คนขับดูสถานะงาน

คนขับเห็นข้อมูลหลัก:

- ทะเบียนรถ
- จำนวนคนงานที่ต้องใช้
- ตลาดทั้งหมด
- แผงทั้งหมด
- สินค้าตามตั๋ว
- สถานะรถ
- สถานะพร้อมลงหรือรอ

### 5.3 รถยังไม่พร้อมลง

กรณีรถหรือปลายทางยังไม่พร้อม:

1. งานรถอยู่ใน `WAIT`
2. ตลาดและตั๋วที่เกี่ยวข้องอยู่ในสถานะรอ
3. ร้านค้า/แม่ค้าอาจได้รับแจ้งว่ารถมาถึงแล้ว
4. คนขับและร้านค้าประสานความพร้อม
5. เมื่อพร้อม คนขับกด `พร้อมลง` หรือ `ลงเลย`

### 5.4 คนขับกดพร้อมลง

เมื่อคนขับกดพร้อมลง:

1. ระบบตรวจว่างานยังอยู่ในสถานะที่กดพร้อมลงได้
2. ระบบเปลี่ยน `VehicleJob` เป็น `DISPATCH_NOW`
3. ระบบเปลี่ยน `MarketJob` ทุกตลาดที่ยังไม่จบเป็น `DISPATCH_NOW`
4. ระบบเปลี่ยน `GateTicket` ที่ยังไม่จบเป็น `READY`
5. ระบบเริ่มกระบวนการเรียกคนงาน
6. Dashboard และ Client ได้รับสถานะใหม่

### 5.5 คนขับกดพร้อมลงซ้ำ

ถ้างานเป็น `DISPATCH_NOW` อยู่แล้ว:

1. ระบบไม่สร้างงานใหม่
2. ระบบไม่ reset คิว
3. ระบบตอบว่ารถพร้อมลงอยู่แล้ว

ถ้างานเป็น `IN_PROGRESS`, `COMPLETED` หรือ `CANCELLED`:

1. ระบบไม่อนุญาตให้กดพร้อมลง
2. ระบบแจ้งสถานะปัจจุบันกลับไป

## 6. Flow คิวคนงาน

### 6.1 คนงาน Login

1. คนงานกรอกรหัสหรือข้อมูลยืนยันตัวตน
2. ระบบตรวจบัญชีคนงาน
3. ระบบตรวจ device identity
4. ระบบบันทึกหรืออัปเดต push token
5. คนงานเข้าสู่ระบบได้ แม้ยังไม่กด Online

การ Login ไม่เท่ากับพร้อมรับงาน

### 6.2 คนงานกด Online

เมื่อคนงานกด Online:

1. ระบบตรวจบัญชีว่ายัง active
2. ระบบตรวจสิทธิ์ทำงานตามกะหรือเวลาที่ได้รับอนุมัติขยาย
3. ระบบตรวจว่าไม่มี active assignment
4. ระบบใส่คนงานเข้าคิว FIFO
5. สถานะคนงานเป็น `ready`
6. Dashboard เห็นจำนวนคนในคิวเพิ่มขึ้น
7. ระบบอาจ trigger dispatch งานที่รอคนงานอยู่

### 6.3 FIFO queue

กฎ FIFO:

1. คนที่เข้าคิวก่อนถูกเรียกก่อน
2. คนงานหนึ่งคนต้องไม่อยู่ในคิวซ้ำหลายตำแหน่ง
3. เมื่อถูกดึงไป assign งาน สถานะเปลี่ยนเป็น `busy`
4. เมื่อจบงานและยังมีสิทธิ์ทำงาน สามารถกลับเข้าคิวได้
5. ถ้าหมดสิทธิ์ทำงาน ระบบไม่ควร auto-ready

### 6.4 คนงานพัก

คนงานพักได้เฉพาะตอน `ready`

ขั้นตอน:

1. คนงานกดพัก
2. ระบบตรวจว่ายังอยู่ในสถานะ `ready`
3. ระบบนำออกจาก FIFO
4. สถานะเป็น `break`
5. ระบบตั้งเวลาพัก
6. เมื่อครบเวลา ระบบพิจารณากลับเข้าคิว

กรณีพักไม่ได้:

- มีงานอยู่
- offline
- อยู่ใน break แล้ว
- เกินจำนวนพักที่กำหนด
- หมดสิทธิ์ทำงานตามกะ

### 6.5 กลับจากพัก

เมื่อครบเวลาพัก:

1. ระบบตรวจว่าคนงานยัง active
2. ระบบตรวจว่ายังอยู่ในสถานะ `break`
3. ระบบตรวจสิทธิ์ทำงาน
4. ถ้าผ่านทั้งหมด ให้กลับเข้า FIFO เป็น `ready`
5. ถ้าไม่ผ่าน ให้เป็น `offline`

## 7. Flow Dispatch คนงาน

### 7.1 เงื่อนไขเริ่ม dispatch

ระบบ dispatch เมื่อ:

1. งานรถอยู่ใน `DISPATCH_NOW`
2. งานยังต้องการคนงานเพิ่ม
3. มีคนงาน `ready` ใน FIFO
4. คนงานมีสิทธิ์ทำงาน

ระบบไม่ dispatch งานรถที่:

- `WAIT`
- `COMPLETED`
- `CANCELLED`
- หรือ `IN_PROGRESS` จาก background sweep ถ้าเป็นการเติมคนอัตโนมัติที่ไม่ต้องการตาม policy

### 7.2 จำนวนคนงานที่ต้องเรียก

จำนวนที่ต้องเรียก:

```text
workersNeeded = workersRequired - activeAssignments
```

active assignment คือ assignment ที่ยังอยู่ในสถานะ:

- `PENDING`
- `ACCEPTED`
- `SCANNED`
- `COUNTING`

### 7.3 การมอบหมายงานให้คนงาน

สำหรับคนงานแต่ละคน:

1. ระบบ pop คนงานจาก FIFO
2. เปลี่ยนสถานะคนงานเป็น `busy`
3. สร้าง assignment ระดับรถ
4. assignment เริ่มที่ `PENDING`
5. ระบบส่ง push notification ให้คนงาน
6. ระบบส่ง event เฉพาะห้องของคนงานคนนั้น
7. ระบบตั้ง timer รับงาน 60 วินาที

### 7.4 คนงานไม่พอ

ถ้าคนงานในคิวไม่พอ:

1. ระบบ assign ได้เท่าที่มี
2. งานรถยังอยู่ในกลุ่มพร้อม dispatch
3. Dashboard แสดงว่า `รอคิวลง` หรือยังรอคนงานเพิ่ม
4. เมื่อคนงานใหม่ Online ระบบ dispatch เพิ่มตามจำนวนที่ขาด

### 7.5 ป้องกันการเรียกคนเกิน

ระบบต้องป้องกัน:

1. dispatch ซ้อนจากหลาย process
2. assignment ซ้ำให้คนงานคนเดิมในรถเดียวกัน
3. คนงานคนเดียวถูก assign ให้หลายงาน active พร้อมกัน
4. คนงานที่หมดสิทธิ์ทำงานถูก dispatch
5. งานที่ครบจำนวนคนแล้วถูกเติมคนเพิ่ม

## 8. Flow การรับงานของคนงาน

### 8.1 คนงานได้รับงาน

เมื่อคนงานได้รับงาน:

1. Mobile แสดงข้อมูลงาน
2. คนงานเห็นเวลานับถอยหลัง 60 วินาที
3. คนงานเลือก `รับงาน` หรือ `ปฏิเสธ`

### 8.2 คนงานกดรับงาน

เมื่อกดรับงาน:

1. ระบบตรวจ assignment เป็นของคนงานคนนั้น
2. ระบบตรวจ assignment ยังเป็น `PENDING`
3. ระบบตรวจสิทธิ์ทำงาน
4. ระบบเปลี่ยน assignment เป็น `ACCEPTED`
5. ระบบบันทึกเวลารับงาน
6. ระบบยกเลิก timer รับงาน
7. ระบบตั้ง timer scan QR 15 นาที
8. ระบบบันทึก deadline การ scan
9. Mobile แสดงว่ารอสแกน QR

### 8.3 คนงานปฏิเสธงาน

เมื่อปฏิเสธ:

1. ระบบตรวจ assignment เป็นของคนงานคนนั้น
2. ระบบตรวจ assignment ยังเป็น `PENDING`
3. ระบบเปลี่ยน assignment เป็น `REJECTED`
4. ระบบตั้งคนงานเป็น `offline`
5. ระบบเรียกคนงานทดแทน
6. Dashboard เห็นสถานะ assignment ถูกปฏิเสธ

### 8.4 คนงานไม่ตอบรับภายใน 60 วินาที

เมื่อหมดเวลา:

1. timer ทำงาน
2. ระบบตรวจว่า assignment ยังเป็น `PENDING`
3. ระบบเปลี่ยน assignment เป็น `TIMEOUT`
4. ระบบตั้งคนงานเป็น `offline`
5. ระบบเรียกคนงานทดแทน
6. ถ้า assignment ถูกกดรับไปแล้วก่อน timer ทำงาน ระบบต้องไม่ลงโทษซ้ำ

### 8.5 รับงานซ้ำ

ถ้าคนงานกดรับซ้ำหลังรับไปแล้ว:

1. ระบบไม่ควรสร้าง assignment ใหม่
2. ระบบตอบตามสถานะปัจจุบัน
3. ถ้าสถานะไม่ใช่ `PENDING` ต้องไม่เปลี่ยนเป็น `ACCEPTED` ซ้ำโดยไม่มีเงื่อนไข

## 9. Flow การสแกน QR ของคนงาน

### 9.1 QR คนงาน

QR คนงานเป็น QR ชุดกลางของรถหรือกลุ่มตลาด

คุณสมบัติ:

1. ผูกกับ `VehicleJob`
2. อาจผูกกับ batch ของ QR ชุดเดียวกัน
3. อาจมี token หลายตัวตามตั๋ว แต่ต้องอยู่ในชุดเดียวกัน
4. คนงานสแกนจากตั๋วใบใดก็ได้ในชุดนั้น
5. token ต้องมีวันหมดอายุ

### 9.2 คนงานสแกน QR สำเร็จ

เมื่อคนงานสแกน:

1. ระบบตรวจ assignment เป็นของคนงาน
2. ระบบตรวจ assignment เป็น `ACCEPTED`
3. ระบบตรวจว่ายังไม่เกิน deadline scan
4. ระบบตรวจ QR token ว่าถูกต้อง
5. ระบบตรวจ QR token ผูกกับรถเดียวกับ assignment
6. ระบบตรวจ QR token อยู่ใน QR set ของรถ
7. ระบบเปลี่ยน assignment เป็น `SCANNED`
8. ระบบบันทึกเวลาสแกน
9. ระบบยกเลิก timer scan ของคนงานคนนั้น
10. ระบบตรวจว่าคนงานครบตาม `workersRequired` และทุกคน scan แล้วหรือยัง

### 9.3 ยังสแกนไม่ครบ

ถ้ายังมีคนงานที่ยังไม่สแกน:

1. assignment ของคนที่สแกนแล้วเป็น `SCANNED`
2. งานรถยังไม่เริ่มจริง
3. งานยังไม่อนุญาตให้ส่งยอด
4. Mobile ของคนที่สแกนแล้วแสดงว่ารอคนงานครบ
5. Dashboard แสดงสถานะรอสแกนครบ

### 9.4 สแกนครบทุกคน

เมื่อ active assignments ครบ `workersRequired` และทุกคนสแกนแล้ว:

1. ระบบเปลี่ยน `VehicleJob` เป็น `IN_PROGRESS`
2. ระบบเปลี่ยน `MarketJob` ทุกตลาดที่ยังไม่จบเป็น `IN_PROGRESS`
3. ระบบเปลี่ยน `GateTicket` ทุกตั๋วที่ยังไม่จบเป็น `IN_PROGRESS`
4. ระบบเริ่มช่วงเวลาทำงานของคนงานทุกคนในชุด
5. ระบบอนุญาตให้ส่งยอดสินค้าได้
6. Dashboard และ Mobile แสดงว่างานเริ่มแล้ว

### 9.5 ไม่สแกนภายใน 15 นาที

เมื่อหมดเวลา scan:

1. timer ทำงาน
2. ระบบตรวจว่า assignment ยังเป็น `ACCEPTED`
3. ระบบเปลี่ยน assignment เป็น `TIMEOUT`
4. ระบบตั้งคนงานเป็น `offline`
5. ระบบเรียกคนงานทดแทน
6. assignment ที่ scan ไปแล้วต้องไม่ถูก timeout ซ้ำ

### 9.6 QR ผิดชุดหรือหมดอายุ

ถ้า QR token ไม่ถูกต้อง:

1. ระบบปฏิเสธการ scan
2. assignment ยังไม่เปลี่ยนเป็น `SCANNED`
3. timer scan ยังเดินต่อจนกว่าจะ scan สำเร็จหรือหมดเวลา

กรณีที่ถูกปฏิเสธ:

- token ไม่มี
- token ปลอม
- token หมดอายุ
- token อยู่คนละรถ
- token ไม่อยู่ใน QR set ของรถ
- assignment ไม่ใช่ของคนงานคนนั้น
- assignment ไม่ได้อยู่ในสถานะ `ACCEPTED`

### 9.7 Admin ต่อเวลา scan

Admin ต่อเวลา scan ได้เฉพาะ assignment ที่:

- อยู่ในสถานะ `ACCEPTED`
- ยังไม่ได้ scan
- งานยังไม่ terminal

เมื่อ Admin ต่อเวลา:

1. ระบบอัปเดต deadline scan
2. ระบบเลื่อน timer
3. ระบบบันทึกเหตุผล
4. ระบบแจ้ง Dashboard และ Mobile ตาม contract

## 10. Flow การลงสินค้าและส่งยอด

### 10.1 เงื่อนไขก่อนส่งยอด

คนงานคนใดก็ได้ในชุด assignment ของรถนั้นสามารถส่งยอดของตั๋ว/แผงได้ ไม่จำเป็นต้องเป็นคนงานคนใดคนหนึ่งโดยเฉพาะ เพราะงานลงสินค้าใช้คนงาน 1 ชุดร่วมกันระดับรถ แต่ระบบต้องตรวจว่าคนที่กดส่งเป็นคนงานที่อยู่ใน active assignment ของรถคันนั้นจริง

คนงานส่งยอดได้เมื่อ:

1. assignment เป็น `SCANNED` หรือ `COUNTING`
2. งานรถเป็น `IN_PROGRESS`
3. สินค้าที่ส่งยอดอยู่ในรถคันนั้นจริง
4. สินค้าที่ส่งยอดอยู่ในตั๋ว/แผงจริง
5. payload ไม่มี product ซ้ำผิดกฎในรอบเดียวกัน
6. ตั๋ว/แผงยังไม่เป็น `COMPLETED` หรือ `CANCELLED`

### 10.2 การส่งยอดครั้งแรก

ขั้นตอน:

1. คนงานเลือกตั๋ว/แผง
2. คนงานกรอกจำนวนสินค้าจริง
3. คนงานกดส่งยอด
4. ระบบตรวจสิทธิ์และสถานะ
5. ระบบบันทึกจำนวนสินค้า
6. ระบบเปลี่ยน assignment เป็น `COUNTING`
7. ระบบเปลี่ยนตั๋วที่ถูกส่งยอดเป็น `SUBMITTED`
8. ระบบเปลี่ยน vendor status เป็น `PENDING`
9. ระบบบันทึกเวลาส่งยอด
10. ระบบส่งข้อมูลให้ร้านค้า/แม่ค้าแผงนั้นยืนยัน
11. ระบบอัปเดตสถานะตั๋ว/แผงนั้นให้ Mobile ของคนงานทุกคนในชุดรถเดียวกันเห็นตรงกัน เช่น `ส่งยอดนับแล้ว` และ `รอยืนยัน`

สถานะการส่งยอดเป็นสถานะระดับ `GateTicket` และสินค้าในตั๋ว ไม่ใช่สถานะส่วนตัวของหน้าจอคนงานคนใดคนหนึ่ง ดังนั้นเมื่อมีคนหนึ่งกดส่งแล้ว คนงานคนอื่นที่เปิดงานรถ/ตลาด/แผงเดียวกันต้องเห็นสถานะเปลี่ยนตาม

### 10.3 ส่งยอดหลายแผง

คนงานชุดเดียวกันสามารถส่งยอดแยกตามแผง

กฎ:

1. แผงหนึ่งส่งยอดแล้ว ไม่จำเป็นต้องรอแผงอื่นก่อน
2. แต่ละแผงมีสถานะยืนยันของตัวเอง
3. รถยังไม่จบจนกว่าทุกแผงในทุกตลาดจบ
4. ตลาดยังไม่จบจนกว่าทุกแผงในตลาดนั้นจบ
5. ถ้าคนงานคนหนึ่งส่งยอดแผง A แล้ว คนงานคนอื่นในรถเดียวกันต้องเห็นแผง A เป็น `SUBMITTED`/`รอยืนยัน` และไม่ควรแสดงเหมือนยังไม่ส่ง
6. แผง B, C หรือแผงอื่นที่ยังไม่ส่งยังคงเป็นสถานะของตัวเอง ไม่ถูกปิดตามแผง A

### 10.4 แก้ไขยอด

แก้ไขยอดได้เมื่อ:

1. ร้านค้าปฏิเสธยอด
2. ตั๋วเข้าสู่ `NEEDS_REVISION`
3. vendor status เป็น `REJECTED`
4. assignment ของคนงานอยู่ในสถานะที่อนุญาตให้ส่งใหม่

เมื่อส่งใหม่:

1. ระบบบันทึกยอดใหม่
2. ระบบเปลี่ยนตั๋วกลับเป็น `SUBMITTED`
3. vendor status กลับเป็น `PENDING`
4. ระบบส่งยอดใหม่ให้ร้านค้าตรวจอีกครั้ง
5. Mobile ของคนงานทุกคนในชุดรถเดียวกันต้องเปลี่ยนจาก `รอแก้ไขยอด` กลับเป็น `ส่งยอดนับแล้ว`/`รอยืนยัน` สำหรับตั๋ว/แผงนั้น

### 10.5 ส่งยอดผิดสินค้า

ถ้าคนงานส่ง product ที่ไม่อยู่ในรถหรือตั๋ว:

1. ระบบปฏิเสธ request
2. ไม่เปลี่ยนสถานะ assignment
3. ไม่เปลี่ยนสถานะตั๋ว
4. ไม่ส่งข้อความให้ร้านค้า

### 10.6 ส่งยอดโดยคนงานผิดคน

ถ้าคนงานพยายามส่งยอด assignment ของคนอื่น:

1. ระบบปฏิเสธ request
2. ไม่บันทึก count
3. ไม่เปลี่ยนสถานะใด ๆ

## 11. Flow ร้านค้ายืนยันหรือปฏิเสธ

### 11.1 ร้านค้าได้รับยอด

ร้านค้าเห็นข้อมูล:

- ตั๋ว/แผง
- รายการสินค้า
- จำนวนที่คาดไว้
- จำนวนที่คนงานนับได้
- ปุ่มยืนยัน
- ปุ่มปฏิเสธพร้อมเหตุผล

### 11.2 ร้านค้ายืนยันยอด

เมื่อร้านค้ายืนยัน:

1. ระบบตรวจตั๋วยังอยู่ในสถานะ `SUBMITTED`
2. ระบบเปลี่ยนตั๋วเป็น `COMPLETED`
3. ระบบเปลี่ยน vendor status เป็น `CONFIRMED`
4. ระบบบันทึกเวลายืนยัน
5. ระบบตรวจว่าตลาดของตั๋วนั้นจบหรือยัง
6. ระบบตรวจว่ารถทั้งคันจบหรือยัง

### 11.3 ยืนยันซ้ำ

ถ้าร้านค้ากดยืนยันซ้ำหลังตั๋วเป็น `COMPLETED`:

1. ระบบตอบแบบ idempotent
2. ไม่สร้าง side effect ซ้ำ
3. ไม่คืนคนงานซ้ำ
4. ไม่ส่ง event completion ซ้ำโดยไม่มีเหตุ

### 11.4 ร้านค้าปฏิเสธยอด

เมื่อร้านค้าปฏิเสธ:

1. ระบบตรวจว่าตั๋วอยู่ในสถานะที่ปฏิเสธได้
2. ระบบเปลี่ยนตั๋วเป็น `NEEDS_REVISION`
3. ระบบเปลี่ยน vendor status เป็น `REJECTED`
4. ระบบบันทึกเหตุผล
5. ระบบเปิดให้คนงานส่งยอดใหม่
6. ระบบแจ้ง Mobile/Dashboard ว่าต้องแก้ไขยอด

### 11.5 ปฏิเสธหลังตั๋วปิดแล้ว

ถ้าตั๋วเป็น `COMPLETED` แล้ว:

1. ระบบไม่ควรอนุญาตให้ปฏิเสธตาม flow ปกติ
2. ถ้าจำเป็นต้องแก้ไขภายหลัง ต้องใช้ admin override หรือ flow แก้ไขเอกสารที่แยกจากงานลงสินค้าปกติ

---

## 12. Flow การปิดตลาดและปิดรถ

### 12.1 ปิดตั๋ว

ตั๋วปิดเมื่อ:

- ร้านค้ายืนยันยอดแล้ว
- หรือ Admin ยกเลิกตั๋วนั้น

สถานะ terminal ของตั๋ว:

- `COMPLETED`
- `CANCELLED`

### 12.2 ปิดตลาด

ตลาดปิดเมื่อทุกตั๋วในตลาดเป็น terminal

ตัวอย่าง:

| ตั๋ว | สถานะ |
|---|---|
| A1 | `COMPLETED` |
| A2 | `COMPLETED` |
| A3 | `CANCELLED` |

ตลาดนี้ปิดได้ เพราะทุกตั๋ว terminal

ถ้ายังมีตั๋ว `IN_PROGRESS`, `SUBMITTED` หรือ `NEEDS_REVISION` ตลาดยังไม่ปิด

### 12.3 ปิดรถ

รถปิดเมื่อทุกตลาดในรถเป็น terminal

เมื่อรถปิด:

1. ระบบเปลี่ยน `VehicleJob` เป็น `COMPLETED`
2. ระบบเปลี่ยน active assignment เป็น `COMPLETED`
3. ระบบปิดช่วงเวลาทำงานของคนงาน
4. ระบบ clear timer ที่เกี่ยวข้อง
5. ระบบคืนคนงานเข้าคิวหากยังมีสิทธิ์ทำงาน
6. ระบบส่ง event ให้ Dashboard/Client
7. ระบบดำเนินการ side effects หลังปิดงาน เช่น แจ้งค่าบริการหรือขอ rating ตาม flow ที่กำหนด

### 12.4 รถยังไม่ปิดเมื่อยืนยันบางแผง

ถ้าบางแผงยืนยันแล้วแต่ยังมีแผงอื่นรอ:

1. ตั๋วที่ยืนยันแล้วเป็น `COMPLETED`
2. ตั๋วอื่นยังดำเนินต่อ
3. ตลาดอาจยังไม่จบถ้ายังมีตั๋วไม่ terminal
4. รถยังไม่จบ
5. คนงานยังไม่คืนคิว

## 13. Flow การยกเลิกโดย Admin

### 13.1 หลักการยกเลิก

Admin action สำคัญต้องมีเหตุผล

ข้อมูลที่ควรบันทึก:

- ผู้ดำเนินการ
- role
- scope ที่ยกเลิก
- เหตุผลมาตรฐาน
- เหตุผลข้อความ
- สถานะก่อนและหลัง
- เวลาดำเนินการ

### 13.2 ยกเลิกทั้งรถ

เมื่อ Admin ยกเลิกทั้งรถ:

1. ระบบตรวจรถยังไม่ terminal
2. ระบบเปลี่ยน `VehicleJob` เป็น `CANCELLED`
3. ระบบเปลี่ยน `MarketJob` ที่ยังไม่ terminal เป็น `CANCELLED`
4. ระบบเปลี่ยน `GateTicket` ที่ยังไม่ terminal เป็น `CANCELLED`
5. ระบบไม่เปลี่ยนตั๋วที่ `COMPLETED`
6. ระบบยกเลิก active assignments
7. ระบบ clear timers
8. ระบบคืนคนงานตาม policy
9. ระบบบันทึก audit
10. ระบบแจ้ง Dashboard/Client

### 13.3 ยกเลิกตลาด

เมื่อ Admin ยกเลิกตลาด:

1. ระบบตรวจตลาดยังไม่ terminal
2. ระบบเปลี่ยน `MarketJob` เป็น `CANCELLED`
3. ระบบเปลี่ยนตั๋วที่ยังไม่ terminal ในตลาดนั้นเป็น `CANCELLED`
4. ระบบไม่เปลี่ยนตั๋วที่ `COMPLETED`
5. ระบบตรวจว่ารถยังมีตลาดหรือตั๋ว active เหลือหรือไม่
6. ถ้ายังมี active เหลือ รถยังดำเนินต่อ
7. ถ้าไม่มี active เหลือ รถเป็น terminal ตามกฎ
8. ระบบคืนคนงานเมื่อไม่มีงาน active เหลือในรถ

### 13.4 ยกเลิกตั๋ว/แผง

เมื่อ Admin ยกเลิกตั๋ว:

1. ระบบตรวจตั๋วไม่ใช่ `COMPLETED`
2. ระบบเปลี่ยนตั๋วเป็น `CANCELLED`
3. vendor status เป็น `CANCELLED`
4. ระบบตรวจว่าตลาดจบหรือยัง
5. ระบบตรวจว่ารถจบหรือยัง
6. ถ้ายังมีตั๋ว active อื่น คนงานยังทำงานต่อ
7. ถ้ายกเลิกจนไม่มี active ticket เหลือ ระบบปิด parent และคืนคนงานตาม policy

### 13.5 ยกเลิกคนงานรายคน

เมื่อ Admin ยกเลิก assignment คนงาน:

1. ระบบตรวจ assignment ยัง active
2. ระบบเปลี่ยน assignment เป็น `CANCELLED`
3. ระบบปิด work segment ของคนนั้นถ้ามี
4. ระบบ clear timer ของ assignment นั้น
5. ระบบตั้งคนงานเป็น `offline`
6. ระบบบันทึกเหตุผล
7. ระบบไม่เติมคนทดแทนอัตโนมัติ หาก policy กำหนดให้เป็น manual replacement

### 13.6 เปลี่ยนรถกลับเป็น WAIT

ใช้เมื่อรถเคยพร้อมลงแล้ว แต่ต้องกลับไปรอ

เงื่อนไขทั่วไป:

1. รถยังไม่ terminal
2. รถยังไม่ควรอยู่ในสถานะที่ไม่อนุญาตให้ถอยกลับ
3. Admin ต้องระบุเหตุผล

เมื่อเปลี่ยนกลับ:

1. `VehicleJob` กลับเป็น `WAIT`
2. `MarketJob` ที่ยังไม่จบกลับเป็น `WAIT`
3. `GateTicket` ที่ยังไม่จบกลับเป็น `WAIT`
4. active assignment ถูกยกเลิก
5. timer ถูก clear
6. คนงานถูกคืนคิวหรือปรับสถานะตาม policy
7. Dashboard แสดงรถกลับไปรอ

## 14. Flow การ assign คนงานแบบ Manual

### 14.1 เหตุผลที่ใช้ manual assign

Admin อาจ manual assign เมื่อ:

- คนงานในคิวไม่พอแต่มีคนงานพร้อมเฉพาะราย
- ต้องแก้ปัญหา dispatch ผิดพลาด
- ต้องเติมคนงานหลังยกเลิกคนงานรายคน
- มีเหตุฉุกเฉินในหน้างาน

### 14.2 เงื่อนไขก่อน assign

ระบบต้องตรวจ:

1. งานอยู่ในสถานะที่รับคนงานได้
2. active assignments ยังไม่ครบ `workersRequired`
3. คนงาน active
4. คนงานไม่มี active assignment อื่น
5. คนงานอยู่ในสถานะ `ready`
6. คนงานยังมีสิทธิ์ทำงานตามกะหรือ extension
7. Admin ระบุเหตุผล

### 14.3 หลัง assign

1. คนงานถูกนำออกจาก FIFO
2. สถานะคนงานเป็น `busy`
3. assignment เป็น `PENDING`
4. ระบบตั้ง timer รับงาน 60 วินาที
5. ระบบส่ง notification
6. ระบบบันทึก audit

## 15. Flow Timeout และ Recovery

### 15.1 Acceptance timeout

เกิดเมื่อคนงานไม่กดรับงานภายใน 60 วินาที

ผลลัพธ์:

1. assignment เป็น `TIMEOUT`
2. คนงานเป็น `offline`
3. ระบบเรียกคนงานทดแทน
4. timer ต้อง idempotent

### 15.2 Scan timeout

เกิดเมื่อคนงานรับงานแล้วไม่สแกน QR ภายใน 15 นาที

ผลลัพธ์:

1. assignment เป็น `TIMEOUT`
2. คนงานเป็น `offline`
3. work segment ไม่ควรเริ่มหากยังไม่สแกนครบ
4. ระบบเรียกคนงานทดแทน
5. timer ต้อง idempotent

### 15.3 Timer หายหรือระบบ restart

เมื่อระบบเริ่มทำงานใหม่:

1. ระบบอ่าน active assignments จากฐานข้อมูล
2. ถ้า assignment เป็น `PENDING` ต้องมี acceptance timer
3. ถ้า assignment เป็น `ACCEPTED` ต้องมี scan timer
4. ถ้า timer หายและยังไม่ถึง deadline ให้สร้าง timer ใหม่
5. ถ้าเลย deadline แล้ว ให้เรียก timeout handler
6. ห้าม timeout assignment ที่เปลี่ยนสถานะไปแล้ว

### 15.4 Redis queue/state เพี้ยน

ระบบต้องใช้ฐานข้อมูลเป็น source of truth

กรณีที่ต้อง reconcile:

1. worker อยู่ใน FIFO แต่ inactive
2. worker อยู่ใน FIFO แต่หมดสิทธิ์ทำงาน
3. worker มี active assignment แต่ Redis state ไม่ใช่ `busy`
4. worker มี active assignment แต่ยังอยู่ใน FIFO
5. Redis state เป็น `busy` แต่ไม่มี active assignment

ผลลัพธ์ที่ปลอดภัย:

- active assignment ต้องทำให้ worker เป็น `busy`
- worker ที่ไม่ควรพร้อมต้องถูกนำออกจาก FIFO
- orphan `busy` ที่ระบบไม่มั่นใจควรถูกตั้งเป็น `offline`
- ระบบไม่ควร auto-ready worker ที่สถานะไม่ชัดเจน

## 16. Flow Dashboard และสถานะที่แสดง

### 16.1 Dashboard รถ

Dashboard รถควรแสดง:

- รถ
- ตลาดทั้งหมด
- แผง/ตั๋วทั้งหมด
- สินค้าและจำนวนคาดไว้
- จำนวนคนงานที่ต้องใช้
- คนงานที่ assigned
- สถานะรถ
- สถานะตลาด
- สถานะตั๋ว
- สถานะยืนยันร้านค้า

### 16.2 Dashboard คนงาน

Dashboard คนงานควรแสดง:

- worker code
- ชื่อ
- สีเสื้อ/เบอร์เสื้อ
- สถานะ queue
- assignment ปัจจุบัน
- งานที่กำลังทำ
- เวลาทำงาน
- เวลาพัก
- action ที่ Admin ทำได้

### 16.3 การแสดงสถานะงาน

ตัวอย่าง mapping:

| Backend state | Display |

### 16.4 การแสดงสถานะส่งยอด

| เงื่อนไข | Display |

| ร้านค้าปฏิเสธ | รอแก้ไขยอด |
| ร้านค้ายืนยันแล้ว | ยืนยันแล้ว |

## 17. Flow Notification

### 17.1 Notification ถึงคนงาน

ส่งเมื่อ:

- มีงานใหม่
- งานถูกยกเลิก
- ต้องสแกน QR
- ต้องแก้ไขยอด
- ถูก force logout
- แม่ค้า/ร้านค้าให้คะแนนงานที่คนงานอยู่ในชุดนั้น
- ยอดเงินของรอบก่อนหน้าพร้อมให้ดูตาม policy

### 17.2 Notification ถึงร้านค้า

ส่งเมื่อ:

- รถมาถึง
- มีงานรอความพร้อม
- คนงานส่งยอด
- ต้องยืนยันยอด
- ค่าบริการพร้อมแจ้ง
- ขอ rating

### 17.3 Notification ถึง Admin/Dashboard

ส่งเมื่อ:

- รถใหม่เข้าระบบ
- สถานะรถเปลี่ยน
- สถานะตลาดเปลี่ยน
- สถานะตั๋วเปลี่ยน
- คนงานเข้า/ออกคิว
- assignment เปลี่ยนสถานะ
- timer หมดเวลา
- งานถูกยกเลิก
- งานปิดสำเร็จ

### 17.4 หลักการส่ง notification

1. ต้องไม่ส่งข้อมูลงานของคนงานให้คนอื่น
2. ต้องไม่ส่ง completion side effect ซ้ำเมื่อ confirm ซ้ำ
3. ควรมี idempotency สำหรับ event สำคัญ
4. ควรมีประวัติการส่งและ retry สำหรับ channel สำคัญ

### 17.5 Flow การให้คะแนนจากร้านค้า

การให้คะแนนเกิดหลังงานของตั๋ว/แผงหรืองานรถเข้าสู่จุดที่ร้านค้าสามารถประเมินได้ตาม policy ของระบบ โดยทั่วไปจะเกิดหลังร้านค้ายืนยันยอดหรือหลังรถปิดงานแล้ว

ขั้นตอน:

1. ระบบส่งคำขอให้คะแนนไปยังร้านค้า/แม่ค้า
2. ร้านค้าเปิดหน้าหรือข้อความสำหรับให้คะแนน
3. ร้านค้าเลือกคะแนนและอาจระบุความคิดเห็นเพิ่มเติม
4. ระบบตรวจว่าการให้คะแนนผูกกับงาน, ตลาด, ตั๋ว/แผง หรือรถที่ถูกต้อง
5. ระบบบันทึกคะแนนและความคิดเห็น
6. ระบบส่ง event แบบ realtime ไปยัง Mobile ของคนงานที่อยู่ในชุด assignment ของรถนั้น
7. Mobile ของคนงานแสดงคะแนนใหม่โดยไม่ต้องรอเปิดแอปใหม่หรือ refresh เอง
8. Dashboard/Admin สามารถเห็นคะแนนตาม scope ที่กำหนด เช่น ระดับรถ, ตลาด, แผง, คนงาน หรือรอบงาน

กฎสำคัญ:

1. คะแนนจากร้านค้าต้องผูกกับงานที่เกิดขึ้นจริง
2. ถ้าคนงานหลายคนอยู่ในชุดรถเดียวกัน คะแนนนั้นควรแสดงให้คนงานในชุดนั้นเห็นตาม policy เดียวกัน
3. ถ้ามีการให้คะแนนซ้ำ ระบบต้องกำหนดให้ชัดว่าเป็นการแก้ไขคะแนนเดิมหรือปฏิเสธการให้คะแนนซ้ำ
4. คะแนนต้องส่ง realtime ไปฝั่งคนงาน เช่นผ่าน WebSocket, push, หรือ channel realtime ที่ระบบกำหนด
5. การแสดงคะแนนต้องไม่ทำให้เห็นข้อมูลของงานหรือร้านค้าที่คนงานไม่มีสิทธิ์เห็น

### 17.6 Flow การคำนวณเงินและการดูยอดเงินของคนงาน

สูตรคำนวณค่าบริการและรายได้คนงานยังเป็นกฎธุรกิจที่ต้องสรุปแยกต่างหาก เอกสารนี้จึงไม่กำหนดสูตรคำนวณเงิน แต่กำหนด flow และกฎการแสดงผลที่สรุปแล้ว

หลักการ:

1. ระบบต้องสามารถคำนวณหรือบันทึกยอดเงินของคนงานจากงานที่เสร็จแล้วได้
2. รายได้ต้องผูกกับงานจริง เช่น รถ, ตลาด, ตั๋ว/แผง, assignment, วันที่ทำงาน หรือกะ ตามสูตรที่จะสรุปภายหลัง
3. ต้องป้องกันการคำนวณซ้ำจาก confirm ซ้ำ, notification retry หรือ job completion retry
4. ต้องมีประวัติย้อนหลังให้ตรวจสอบได้เมื่อสูตรคำนวณถูกล็อกแล้ว

กฎการแสดงยอดเงินให้คนงาน:

1. คนงานดูยอดเงินย้อนหลังได้ตั้งแต่ “เมื่อวาน” ลงไป
2. คนงานดูยอดเงินของ “วันนี้” ไม่ได้
3. ถ้าวันปัจจุบันคือวันที่ D หน้าคนงานแสดงยอดได้ถึงวันที่ D-1 เท่านั้น
4. งานที่เกิดหรือปิดในวันนี้ต้องไม่แสดงในยอดเงินวันนี้บน Mobile ของคนงาน
5. เมื่อข้ามวัน ระบบจึงเปิดให้ดูยอดของวันก่อนหน้าได้ตามรอบประมวลผลที่กำหนด

ตัวอย่าง:

| วันที่ปัจจุบัน | วันที่ที่คนงานดูยอดเงินได้ | วันที่ที่ยังดูไม่ได้ |
|---|---|---|
| 1 กรกฎาคม | 30 มิถุนายน และก่อนหน้า | 1 กรกฎาคม |
| 2 กรกฎาคม | 1 กรกฎาคม และก่อนหน้า | 2 กรกฎาคม |

ข้อควรระวัง:

1. การปิดงานวันนี้ไม่ได้แปลว่าคนงานเห็นยอดเงินวันนี้ทันที
2. ถ้างานข้ามวัน ต้องกำหนด business date ให้ชัดเจนว่ายอดอยู่วันใด
3. ถ้ามีการแก้ไขงานย้อนหลัง ต้องมีรายการปรับปรุงหรือ audit ให้ตรวจสอบได้
4. Dashboard/Admin อาจเห็นข้อมูลต่างจาก Mobile คนงานตามสิทธิ์และ policy แต่ Mobile คนงานต้องไม่เห็นยอดของวันปัจจุบัน

## 18. กรณีพิเศษและ Edge Cases

### 18.1 Gate ยิงซ้ำ

ถ้า Gate ส่งธุรกรรมเดิมซ้ำ:

1. ระบบตรวจ transaction id หรือ idempotency key
2. ระบบคืนข้อมูลงานเดิม
3. ไม่สร้างรถซ้ำ
4. ไม่ dispatch คนงานซ้ำโดยไม่มีเหตุ

### 18.2 Gate ส่งตั๋วซ้ำใน payload เดียวกัน

ถ้า `gateTicketNo` ซ้ำ:

1. ระบบปฏิเสธ payload
2. ไม่สร้างงานบางส่วน
3. แจ้ง error ให้ Gate

### 18.3 รถพร้อมลงแต่ไม่มีคนงาน

1. รถอยู่ใน `DISPATCH_NOW`
2. Dashboard แสดงรอคิว
3. เมื่อคนงาน Online ระบบ dispatch ต่อ
4. ไม่ต้องให้ Gate ส่งงานใหม่

### 18.4 คนงานรับงานแล้วหมดกะก่อนเริ่ม

1. ระบบตรวจ work eligibility ตอนรับงานและตอนเริ่มงาน
2. ถ้าหมดสิทธิ์ ระบบไม่ควรเริ่มงานให้คนนั้น
3. ระบบต้องจัดการ assignment ตาม policy เช่น timeout/cancel/offline/replace

### 18.5 คนงานสแกนบางคนแล้วอีกคน timeout

1. คนที่สแกนแล้วคงเป็น `SCANNED`
2. คนที่ timeout เป็น `TIMEOUT`
3. งานยังไม่เริ่มถ้ายังไม่ครบ `workersRequired`
4. ระบบเรียกคนงานทดแทน
5. เมื่อคนงานทดแทนรับและสแกนจนครบ จึงเริ่มงาน

### 18.6 ร้านค้าปฏิเสธบางแผง

1. ตั๋วที่ถูกปฏิเสธเป็น `NEEDS_REVISION`
2. ตั๋วอื่นที่ยืนยันแล้วคง `COMPLETED`
3. ตลาดยังไม่ปิดถ้ายังมีตั๋วต้องแก้ไข
4. รถยังไม่ปิด
5. คนงานแก้ไขและส่งยอดใหม่เฉพาะแผงที่ถูกปฏิเสธ

### 18.7 ยกเลิกบางแผงหลังบางแผงยืนยันแล้ว

1. ตั๋วที่ `COMPLETED` ไม่ถูกเปลี่ยน
2. ตั๋วที่ยังไม่จบถูกเปลี่ยนเป็น `CANCELLED`
3. ตลาดปิดถ้าทุกตั๋ว terminal
4. รถปิดถ้าทุกตลาด terminal

### 18.8 Admin ยกเลิกคนงานหลังงานเริ่ม

1. assignment คนนั้นเป็น `CANCELLED`
2. work segment คนนั้นปิด
3. คนงานคนนั้นเป็น `offline`
4. ระบบไม่เติมคนอัตโนมัติถ้า policy กำหนดให้ replacement เป็น manual
5. Admin ต้องตัดสินใจว่าจะ assign คนใหม่หรือให้ชุดที่เหลือทำต่อ

### 18.9 QR คนขับถูกเปิดหลังงานจบ

1. ถ้างาน `COMPLETED` ระบบแสดงว่างานจบแล้ว
2. ไม่อนุญาตให้กดพร้อมลงใหม่
3. ไม่ dispatch คนงานใหม่

### 18.10 QR คนงานถูกใช้ผิดรถ

1. ระบบปฏิเสธ scan
2. assignment ไม่เปลี่ยนสถานะ
3. timer ของ assignment ยังเดินต่อ

## 20. นิยามความจบของงาน

### 20.1 ตั๋วจบ

ตั๋วจบเมื่อ:

- `COMPLETED`
- หรือ `CANCELLED`

### 20.2 ตลาดจบ

ตลาดจบเมื่อ:

- ทุกตั๋วในตลาดเป็น terminal
- และตลาดเปลี่ยนเป็น `COMPLETED` หรือ `CANCELLED`

### 20.3 รถจบ

รถจบเมื่อ:

- ทุกตลาดในรถเป็น terminal
- ทุกตั๋วในทุกตลาดเป็น terminal
- active assignments ถูกปิด
- work segments ถูกปิด
- timers ถูก clear
- คนงานถูกคืนคิวหรือถูกตั้งสถานะตาม policy

### 20.4 งานไม่จบเพียงเพราะ

- คนงานส่งยอดครบ แต่ร้านค้ายังไม่ยืนยัน
- ตลาดหนึ่งยืนยันครบ แต่ตลาดอื่นยังค้าง
- คนงานบางคนสแกนแล้ว แต่ยังไม่ครบชุด
- คนขับกดพร้อมลงแล้ว แต่ยังไม่มีคนงานครบ
- Admin ยกเลิกบางตั๋ว แต่ยังมีตั๋ว active อื่น

## 21. หลักการไม่ให้ flow สับสน

1. Flow หลักยึดรถ ไม่ใช่ตลาดเดี่ยว
2. คนงานถูกเรียกเป็นชุดเดียวต่อรถ
3. เลขที่งาน 14 หลักยังอยู่ระดับตลาด
4. QR คนขับอยู่ระดับรถ
5. QR คนงานเป็นชุดกลาง
6. การเริ่มงานเกิดพร้อมกันทั้งรถหลังคนงานครบและสแกนครบ
7. การยืนยันเกิดแยกระดับแผง
8. การปิดงานเกิดจากล่างขึ้นบน: ตั๋ว -> ตลาด -> รถ
9. `COUNTING` ไม่ควรแปลว่า “กำลังนับ” เสมอไป เพราะอาจหมายถึงส่งยอดแล้วหรือรอแก้ไขยอด
10. `ready` ของคนงานหมายถึงพร้อมรับงานในคิว ไม่ใช่เพียงเปิดแอป
11. `busy` หมายถึงมี assignment หรืออยู่ในกระบวนการทำงาน ไม่ควรอยู่ใน FIFO
12. ถ้าระบบไม่มั่นใจสถานะ worker หลัง incident ให้เลือก `offline` มากกว่า auto-ready
